# GA4 데이터 해석 및 액션 가이드

## 목적

이 문서는 현재 구현된 GA4 데이터가 각각 무엇을 의미하는지와,
실제 운영에서 어떤 액션으로 연결해야 하는지를 정리한 실행 가이드다.

- 이벤트 정의 기준: `docs/analytics/ga4-event-spec.md`
- 리포트 생성 기준: `docs/analytics/ga4-reports-checklist.md`

## 이벤트별 데이터 의미와 활용

| 이벤트              | 데이터 의미                                                 | 바로 할 수 있는 액션                                                            |
| ------------------- | ----------------------------------------------------------- | ------------------------------------------------------------------------------- |
| `job_list_view`     | 사용자가 목록 화면을 조회한 맥락(검색어/패밀리/정렬/페이지) | 조회수 대비 상세 진입(`job_view`)이 낮은 조합을 찾아 목록 카드/정렬 기본값 조정 |
| `job_search`        | 사용자가 실제로 어떤 키워드로 탐색하는지                    | 상위 검색어 중 전환(`apply_success`)이 낮은 키워드는 검색 랭킹/동의어 사전 개선 |
| `job_filter_family` | 카테고리 이동 패턴과 이전/현재 카테고리                     | 자주 이동하는 카테고리를 상단 노출, 거의 쓰이지 않는 카테고리 정리              |
| `job_sort_change`   | 정렬 의도(최신 수정순 vs 등록순)                            | 특정 정렬에서 전환이 높으면 기본 정렬 정책 재검토                               |
| `job_view`          | 어떤 공고가 실제 상세까지 읽히는지                          | 조회 대비 지원 클릭(`apply_click`)이 낮은 공고는 제목/요약/요건 표현 개선       |
| `apply_click`       | 지원 의도 발생 시점과 CTA 위치                              | `cta_source`별 클릭률 비교로 목록 CTA/상세 CTA 우선순위 결정                    |
| `apply_success`     | 실제 지원 완료(핵심 전환)                                   | 채널/로케일/검색어 기준으로 고성과 세그먼트에 예산/운영 집중                    |
| `apply_error`       | 지원 실패 상황 발생                                         | 오류율 임계치(예: 5분 평균 > 2%) 초과 시 즉시 장애 대응 트리거                  |
| `auth_modal_open`   | 인증 요구가 발생한 진입 지점                                | `entry_point`별 모달 오픈 대비 인증 완료율 낮은 지점 UX 개선                    |
| `login_submit`      | 로그인 시도 시작                                            | 제출량 대비 성공률로 로그인 폼/유효성/오류문구 개선 우선순위 판단               |
| `login_success`     | 로그인 완료                                                 | 로그인 완료 후 `apply_success`까지의 후속 전환 모니터링                         |
| `login_error`       | 로그인 실패                                                 | 오류코드별 비중으로 인증 서버 이슈 vs 사용자 입력 이슈 분리 대응                |
| `signup_start`      | 가입 방법 선택(이메일/소셜)                                 | 선호 채널 비중에 따라 가입 UI 버튼 순서 및 강조 문구 최적화                     |
| `signup_success`    | 회원가입 완료                                               | 가입 직후 지원 전환이 낮으면 온보딩/추천 공고 노출 강화                         |
| `signup_error`      | 가입 실패                                                   | 이메일 중복/기타 오류 비중에 따라 메시지 개선 또는 백엔드 점검                  |

## 파라미터별 해석 가이드

| 파라미터             | 값 예시              | 해석                     | 액션 예시                                            |
| -------------------- | -------------------- | ------------------------ | ---------------------------------------------------- |
| `locale`             | `ko`                 | 사용자 언어 컨텍스트     | 특정 로케일 전환 저하 시 번역/카피/현지화 UX 개선    |
| `page_path`          | `/jobs?search=react` | 이벤트 발생 화면/상태    | 특정 경로에서 이탈이 높으면 해당 화면만 우선 개선    |
| `is_authenticated`   | `false`              | 인증 여부                | 비인증 비율이 높은 구간에 로그인 유도 방식 개선      |
| `user_role`          | `candidate`          | 사용자 역할 힌트         | 비후보자 트래픽이 많은 경우 경로 분기/권한 안내 보강 |
| `search_term`        | `backend`            | 사용자의 탐색 의도       | 상위 키워드 전환률 기반으로 추천/자동완성 개선       |
| `family_id`          | `engineering`        | 현재 카테고리 컨텍스트   | 카테고리별 성과 비교로 채용 운영 우선순위 설정       |
| `selected_family_id` | `design`             | 사용자가 클릭한 카테고리 | 클릭 집중 카테고리 중심으로 탭 정보구조 재배치       |
| `previous_family_id` | `engineering`        | 이동 이전 카테고리       | 잦은 왕복 이동 시 카테고리 정의/라벨 개선            |
| `sort`               | `created_desc`       | 현재 정렬 기준           | 성과 높은 정렬을 기본값으로 고려                     |
| `previous_sort`      | `updated_desc`       | 정렬 변경 전 상태        | 변경 전후 성과 비교로 정렬 UX 검증                   |
| `page`               | `3`                  | 목록 페이지 위치         | 페이지가 깊어질수록 전환 급감 시 필터/추천 강화      |
| `jd_id`              | `jd-123`             | 공고 식별자              | 공고별 퍼널(조회-클릭-완료)로 콘텐츠/운영 개선       |
| `cta_source`         | `jobs_list`          | CTA 위치                 | 위치별 성과 차이로 버튼 배치/문구 실험               |
| `modal`              | `login`              | 열린 인증 모달 타입      | 가입 유도 과다/부족 여부 판단                        |
| `entry_point`        | `job_detail_apply`   | 인증이 필요한 맥락       | 맥락별 인증 전환율 최적화                            |
| `method`             | `google`             | 로그인/가입 방식         | 채널 선호도 기준으로 소셜 로그인 우선순위 결정       |
| `error_code`         | `apply_failed`       | 실패 원인 분류 키        | 오류 유형별 대응 책임자/플레이북 연결                |

## 운영 액션 플레이북

### 1. 지원 전환 개선

1. `job_view -> apply_click -> apply_success` 퍼널을 `cta_source`, `locale`로 분해한다.
2. `apply_click` 대비 `apply_success` 하락 구간이 큰 조합을 찾는다.
3. 액션:

- `jobs_list`만 낮으면 목록 CTA 문구/상태 처리 우선 개선
- `job_detail`만 낮으면 상세 상단 정보/요건 표현 우선 개선
- 둘 다 낮으면 지원 API 안정성(`apply_error`) 우선 점검

### 2. 탐색 UX 개선

1. `job_search`, `job_filter_family`, `job_sort_change` 이벤트를 주간 집계한다.
2. 검색량 상위 키워드에서 `apply_success`가 낮은 그룹을 찾는다.
3. 액션:

- 검색 결과 랭킹 조정
- 동의어 사전/오탈자 보정 추가
- 카테고리 탭 순서 재배치

### 3. 인증 퍼널 개선

1. `auth_modal_open -> login_submit/signup_start -> login_success/signup_success`를 `entry_point`별로 본다.
2. 진입점별 전환 누수를 비교한다.
3. 액션:

- 특정 진입점에서 submit 비율이 낮으면 모달 카피/버튼 강조 수정
- submit 대비 success가 낮으면 오류 메시지 개선 및 인증 백엔드 점검

### 4. 장애 탐지 및 대응

1. `apply_error`, `login_error`, `signup_error`를 5분 단위로 모니터링한다.
2. 임계치 초과 시 배포/인프라/외부 인증 상태를 즉시 확인한다.
3. 액션:

- 공통 오류코드 급증: 서버 장애 대응
- 특정 방법(`method`)만 급증: 소셜 공급자/설정 점검
- 특정 경로(`page_path`)만 급증: 프론트 회귀 확인

## 주의사항

- 현재 구현은 소셜 로그인/회원가입의 "성공" 이벤트를 클라이언트에서 직접 보장하지 않는다.
  (외부 리다이렉트 플로우 특성)
- 따라서 소셜 성공 분석은 우선 `auth_modal_open`, `login_submit`, `signup_start`와
  후속 행동(`job_view`, `apply_success`)을 조합해 간접 추정한다.
- PII(이메일, 이름, 전화번호)는 전송하지 않는다.
